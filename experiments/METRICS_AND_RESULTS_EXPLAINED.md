# 指标详解 + 各 Phase 改进点/效率/准确性汇总

本文档面向"想看懂这些数字到底意味着什么"的读者。第一部分讲清楚每个指标**怎么算、什么样
算好**；第二部分按 Phase 逐一说明**改了什么、快了多少、准不准**，所有数字都来自真实跑通的
实验（见 `experiments/phaseN/result.md` 和 `FINAL_REPORT.md`）。

---

## 一、指标详解

### 1. AUC（Area Under the ROC Curve）—— 检索器"打分准不准"

**用在哪**：Phase 2 的 GNN backbone（R-GCN/LightGCN/KGAT）训练，衡量"给定一个 user 和一个
item，模型打的分能不能正确分辨这是不是真实的交互"。

**怎么算**：
1. 准备两组边：**正样本**（真实发生过的 user-item 交互）和**负样本**（随机采样的、没有发生过的
   user-item 组合）。
2. 模型对每条边（不管正负）都打一个分数（比如 user 和 item 的 embedding 做点积）。
3. AUC 衡量的是：**随机抽一个正样本和一个负样本，模型给正样本打的分比负样本高的概率是多少**。
   用 `sklearn.metrics.roc_auc_score(真实标签, 模型打分)` 直接算。

**什么样算好**：
- **0.5** = 瞎猜（正负样本分数完全分不开，跟抛硬币一样）
- **1.0** = 完美（所有正样本分数都比所有负样本高）
- 一般来说 **0.7 以上算能用，0.85+ 算很好**，超过 0.95 就非常强了。
- 本次实验：R-GCN 0.76（勉强能用）、KGAT 0.87（很好）、LightGCN 0.96（非常强）。

**局限**：AUC 只衡量"排序对不对"，不衡量"打分的具体数值有没有意义"，也**不直接等于**下游生成
解释的质量好不好——Phase 2 就发现了这两者可能完全脱钩（AUC 差很多，生成质量却一样）。

---

### 2. BERTScore（Precision / Recall / F1）—— 语义有没有对上

**用在哪**：所有 Phase 的生成质量评估，比较"模型生成的解释文本"和"真实的人工解释文本"。

**怎么算**：
1. 用一个预训练的语言模型（这里用 RoBERTa-large）把两句话的**每一个词**都变成一个带上下文的
   向量（同一个词在不同句子里向量可能不同，因为"上下文"不同）。
2. **Recall（召回率）**：对参考答案（真实解释）里的**每一个词**，去候选文本（模型生成的解释）
   里找一个语义最像的词（用向量余弦相似度衡量），把所有词的最佳匹配分数取平均。
   —— 直觉：**真实答案该提到的点，生成的文本里有没有提到**。比如参考答案提到"啤酒、氛围、便宜"，
   生成的文本里这三个概念都能找到语义相近的表达，Recall 就高。
3. **Precision（精确率）**：反过来，对候选文本（生成的解释）里的**每一个词**，去参考答案里找
   最像的词，取平均。
   —— 直觉：**生成的文本里说的每一点，是不是都有真实依据（参考答案里真提到过）**，而不是
   自己编的、跟真实理由无关的内容。比如生成的文本提到"啤酒、音乐、氛围、干净"，但参考答案
   只支持"啤酒、氛围"，多出来的"音乐、干净"如果在参考答案里找不到语义相近的词，Precision
   就会被拉低。
4. **F1** = Precision 和 Recall 的调和平均，综合两边。
5. 额外做了 **rescale_with_baseline**（用一批"随机句子对"的平均相似度做基线，减掉这个基线再
   归一化），因为不做这一步的话原始余弦相似度普遍在 0.85-0.95 之间挤在一起，看不出区分度。

**什么样算好**（rescale 之后的范围）：
- 理论范围大约 -1 到 1，但实际这类任务里常见范围是 **0.2 - 0.5**。
- **越高越好**。同一批数据里比较，差 0.02-0.03 就算有意义的差距（本次 Phase 1 的
  Power-Link vs Dijkstra 差距就是这个量级：BERT-F1 +0.024）。
- 论文原文汇报的 G-Refer 基线 BERT-F1 是 0.4003，我们复现出 0.4038，这个量级下 0.40+ 算是
  这个任务里"不错"的水平。

### 3. BARTScore —— "生成的文本能不能顺理成章地对应到参考答案"

**用在哪**：所有 Phase 的生成质量评估。

**怎么算**：用一个预训练的 BART（一种"编码-解码"结构的生成式语言模型，这里用
`facebook/bart-large-cnn`）当打分器：**把候选文本当输入，看这个 BART 模型生成参考答案的
概率有多高**（用对数似然 log-probability 衡量）。直觉上：如果候选文本已经把意思说对了，
BART 模型"接着编"出参考答案的原文应该比较容易（概率高）；如果候选文本文不对题，BART 模型
"编"出参考答案就会很困难（概率低）。

**什么样算好**：
- 因为算的是对数概率（log-probability），**数值永远是负数**，**越接近 0 越好**（比如 -2.9
  比 -3.1 好）。
- 没有绝对的"好坏"分界线，主要看**相对比较**：同一批数据下谁的分数更接近 0，谁更好。
- 本次实验里，Power-Link（-2.97）比 Dijkstra（-3.05）更接近 0，说明 Power-Link 生成的解释
  更容易"对应"到真实答案。

---

### 4. BLEURT —— 专门训出来"像不像人类打分"的模型

**用在哪**：所有 Phase 的生成质量评估。

**是什么**：Google 发布的一个**预训练模型**（不是我们自己训的）——本质上是一个 BERT，先在
大量"人工合成的相似句子对+自动打分"数据上预训练，再在**真人评分**过的句子对数据集（WMT
机器翻译质量评测的人工评分）上专门微调过，让它学会"人类觉得这两句话有多像"这件事。

**怎么用**：直接把训练好的权重加载进来，输入 (参考答案, 候选文本) 这一对句子，模型直接输出
一个分数，不需要额外训练。**本次用的是 `bleurt-base-128`（Google 官方发布的权重，从
`storage.googleapis.com` 直接下载），不是自己训的**。

**什么样算好**：
- 大致校准过的范围：**接近 1.0** = 几乎是同一句话（人类会觉得几乎一样，比如完全相同的句子
  实测得分 0.987）；**0 附近** = 中性、不太相关；**明显负数** = 内容基本无关（完全不相关的
  两句话实测得分 -1.674）。
- **越高（越接近 1，越不负）越好**。
- 本次实验分数普遍在 -0.3 到 -0.4 之间，属于这类"用检索信息生成解释"任务的正常区间
  （不是满分句子复述，但也不是毫不相关）。

---

### 5. USR（Unique Sentence Ratio，独特句子占比）—— 是不是在"复读"

**用在哪**：所有 Phase，检查模型是不是在偷懒生成套话。

**怎么算**：把所有生成的解释文本收集起来，看这些句子里有多少是**完全不重复**的（逐字逐词
比较），除以总句子数。

**什么样算好**：
- 范围 0 到 1，**越接近 1 越好**——说明模型是"看情况生成不同的解释"，而不是无论输入什么都
  吐出同一句"这家店很不错，推荐"这种万能套话。
- 本次所有实验结果都是 **1.0000**（500 条测试样本里没有两条完全一样的解释），说明这方面
  没有问题，不是本次对比的关键变量。

## 二、各 Phase 改进点 / 时间效率 / 结果准确性

### Phase 0 — 基线复现

- **改进点**：跑通原始 G-Refer 全流程（检索已有真实数据 → RAFT 微调 Llama-3-8B → 推理 →
  评估），顺手修了 4 个真实 bug（训练保存崩溃、encoder/checkpoint 不匹配、环境里 dgl 完全
  装不上）。
- **时间效率**：训练一次完整 2 epoch 大约 50 小时（2×3090，因为 3090 显存有限，batch/序列
  长度都比论文小）；单样本推理约 0.6 秒。
- **结果准确性**（3000 条真实测试样本，vs 论文 Table 1）：

  | 指标 | 论文 | 复现 | 差距 |
  |---|---|---|---|
  | BERT-F1 | 0.4003 | 0.4038 | +0.0035（几乎一致） |
  | BARTScore | -3.6448 | -2.8901 | 更好 |
  | BLEURT | -0.1336 | -0.3217 | 差了不少（本次唯一有明显差距的指标，详见 phase0 report） |
  | USR | 1.0 | 1.0 | 一致 |

  复现基本成功，BLEURT 偏低可能是用了较小的 BLEURT 变体 + 显存受限导致训练强度不及论文。

### Phase 1 — Power-Link 图幂法替换 Dijkstra 路径检索

- **改进点**：把"逐个 (user, item) pair 用 Dijkstra/Yen's 算法搜最短路径"换成"矩阵幂迭代"
  （用户-物品图的加权邻接矩阵反复自乘，按路径长度归一化选出候选路径），数学上更适合并行、
  不需要逐条搜索。
- **时间效率**：
  - 典型情况：Dijkstra 均值 2.86 秒/对，Power-Link 均值 0.71 秒/对，**快 4 倍**。
  - 极端情况（长尾）：Dijkstra 最坏单个 pair 要 **113 秒**，Power-Link 同样的 pair 只要
    1.6 秒，**快 69 倍**——这正是原方法"某些复杂图结构会让最短路搜索退化很慢"的问题。
- **结果准确性**（500 条真实测试样本，同一个真实 Llama-3-8B）：

  | 指标 | Dijkstra | Power-Link |
  |---|---|---|
  | BERT-F1 | 0.3586 | **0.3822**（更高） |
  | BARTScore | -3.051 | **-2.969**（更好） |
  | BLEURT | -0.476 | **-0.420**（更好） |

  **不仅快，而且准——全部指标都更好**，是本次最扎实的正向结果之一。

### Phase 2 — GNN backbone 换血（R-GCN → LightGCN / KGAT）

- **改进点**：把节点表征的计算方式从 R-GCN（带非线性特征变换的邻居聚合）换成 LightGCN（去掉
  非线性和特征变换，只保留邻居平均）或 KGAT（加注意力机制区分邻居重要性）。
- **时间效率**：
  - 训练收敛速度：LightGCN 在第 10 个 epoch 就追平 R-GCN/KGAT 跑满 100 epoch 的水平。
  - 检索延迟（用新 backbone 重新打分检索路径）：LightGCN 最快（0.526 秒/条），R-GCN 其次
    （0.675 秒），KGAT 最慢（1.162 秒，注意力机制计算开销更大）。
- **结果准确性**：
  - **链接预测 AUC**（backbone 自己的任务）：R-GCN 0.76 → KGAT 0.87 → LightGCN **0.96**，
    差距巨大。
  - **但下游生成质量**（500 条真实测试样本）：三者的 BERT-F1 都在 0.3823-0.3827 之间，
    BARTScore/BLEURT 差异都在小数点第三位——**几乎没有差别**。
  - **这是本次一个重要的诚实发现**：AUC 差距很大，不代表最终生成的解释文本质量也会跟着差
    很多——至少在本次的检索管线（合成图 + 只取 2 条路径）下没有传导过去。仍然推荐 LightGCN
    （效率最优 + AUC 最高），只是不能拿"生成质量"当额外的采纳理由。

### Phase 3 — 子图检索 + Soft-Prompt 注入 LLM

- **改进点**：不再只把检索到的路径/节点"翻译成文字"塞进 prompt，而是取 (user, item) 周围
  的 l-hop 子图，用已训好的 GNN 编码成一个向量，过一个小的投影层（MLP），**当成一个额外的
  "软 token"直接拼接进 LLM 的输入向量序列**（而不是变成文字）。
- **时间效率**：跟 baseline（不用 soft-prompt）几乎一样快，只多了一次子图向量的投影计算，
  可以忽略不计。真正的效率瓶颈在"提前算好这些子图向量"这一步——单个 (user, item) pair 抽子图
  本身就比较慢，所以只在 5000/74212 条训练样本上做了真实抽取（约 6.7%），没有跑全量。
- **结果准确性**（公平对照：同样的 4300 条训练数据、同样的训练轮数，只有"加不加 soft-prompt"
  这一个变量不同）：

  | 指标 | 不加 soft-prompt | 加 soft-prompt |
  |---|---|---|
  | BERT-F1 | **0.3932** | 0.3623（更差） |
  | BARTScore | -3.118 | -3.063（略好） |
  | BLEURT | **-0.325** | -0.382（更差） |

  **诚实的负向结果**：在这个规模下（小模型 TinyLlama、小数据量、只训 3 轮），单独加
  soft-prompt 没有带来提升，4/5 个指标反而变差了。可能是训练数据/轮数不够让模型学会怎么
  利用这个额外信号，也可能是子图向量本身信息量有限（合成图关系退化的老问题）。

### Phase 4 — GraphLoRA（结构感知的 LoRA 微调）

- **改进点**：不是简单地"多塞一个 token"，而是让 LoRA 微调本身的更新幅度**跟着每个样本的
  子图信号动态调整**——每个 (user, item) 样本有自己的一个"门控值"，训练刚开始这个门控值
  严格等于 1（跟普通 LoRA 完全一样），随着训练进行才慢慢学出差异化的调整幅度。
- **时间效率**：跟 Phase 3 的 soft-prompt 版本几乎一样快（门控只是训练时多一次很小的向量
  运算），推理阶段完全没有额外开销——因为训练结束保存模型的时候，LoRA 的更新量已经被"烧录"
  进主干权重里了，门控这个机制本身不需要在推理时重建。
- **结果准确性**（同样公平对照：跟 Phase 3 完全一样的数据/训练轮数，只是 LoRA 是否带门控
  不同）：

  | 指标 | 不加 soft-prompt | 只加 soft-prompt | soft-prompt + GraphLoRA |
  |---|---|---|---|
  | BERT-F1 | 0.3932 | 0.3623 | **0.3958**（最高） |
  | BARTScore | -3.118 | -3.063 | **-2.925**（最好） |
  | BLEURT | -0.325 | -0.382 | **-0.298**（最好） |
  | 训练 eval loss | 0.726 | 0.746 | **0.695**（最低） |

  **本次证据最扎实的正向结果**：GraphLoRA 不仅修好了 Phase 3 单独用 soft-prompt 时"越训越差"
  的问题，还反超了完全不用任何辅助信号的 baseline，在 5 个指标里 4 个最优，连训练 loss 都是
  三者中最低的——三条线索互相印证，不是巧合。

### Phase 5 — 可微 Top-K（DFTopK）+ 排序一致性指标（RCS）

- **改进点**：
  1. 实现了一个"软性"的 top-k 选择函数（用 sigmoid 在候选边界处做平滑），理论上可以替换掉
     "选前 k 个"这种不可求导的硬操作，为将来做"检索器和生成模型端到端联合训练"打基础。
  2. 实现了 RCS（排序一致性分数）：衡量"两个打分模型对同一批候选的排序像不像"，用两个角度
     衡量——**top-k 重合率**（两边选出的前 k 个候选有多少是同一批）和 **Kendall's tau**
     （更细粒度的排序相关系数，-1 到 1，1 表示排序完全一致）。
- **时间效率**：这两个都是轻量级的独立计算组件，不涉及大规模训练/推理，运行时间是秒级的。
- **结果准确性**：
  - DFTopK 的软选择在温度调低时和真正的硬 top-k **重合率 1.000**（完全一致），且反向传播
    时梯度确实能传到打分模型的参数上——组件本身是对的。
  - 用 RCS 去量化 Phase 2 里 R-GCN 和 LightGCN 的排序像不像：top-k 重合率只有 **0.001**，
    Kendall's tau 只有 **0.016**——**两个 AUC 都不错的模型，对同一批候选给出的具体排序几乎
    完全不一样**。这从另一个角度印证了 Phase 2 的发现："模型自己的准确率"和"两个模型之间
    排序有多一致"是两件不同的事，不能互相替代。

---

## 三、一句话总结

跑完全部真实实验后，最值得采纳的三个改进是：**Power-Link**（更快也更准）、**LightGCN**
backbone（效率和自身准确率都最优，虽然没测出下游质量差异）、**GraphLoRA**（唯一在"LLM 微调"
这一层跑出真实正向效果的方案，且修复了单独用 soft-prompt 的负面效果）。子图 soft-prompt
单独使用、以及需要真实多关系图数据的几个方向（IntTower/RippleNet/KGRec/端到端联合训练），
目前证据不支持采纳或因数据缺口无法验证。
