# 粗排 / 精排相关论文泛读清单（详细版）

> **先读这段，理解为什么挑这几篇。**
>
> 工业界的推荐/广告系统，物品动辄上亿，不可能对每个物品都用最贵的模型算一遍。于是拆成**多阶段漏斗**：
>
> ```
> 召回(recall)        亿 → 几千     // 海选，只求"别漏掉好的"，模型极简
> 粗排(pre-ranking)   几千 → 几百   // 初筛，快而糙的打分
> 精排(ranking)       几百 → 几十   // 决赛，最贵最准的模型
> 重排(re-ranking)    几十 → 最终顺序 // 考虑多样性、打散等
> ```
>
> - **粗排 = 用便宜模型做海选，把候选从几千砍到几百**。核心矛盾：又要快、又要准。
> - **精排 = 用最贵模型做决赛，给几百个候选精细打分排序**。
>
> **这跟 G-Refer 有什么关系？** G-Refer 的检索器干的就是同一件事——从一整张用户-商品大图里，一步步筛出最有用的 top-k 路径和节点：
>
> ```
> GNN 给所有边/节点打分        ≈ 召回/粗排的打分
> m-core 剪枝掉高度数热门节点   ≈ 粗筛（砍掉没信息量的候选）
> 边权重排序 + Dijkstra 取 top-k 路径   ≈ 精排（精细选最终的几条）
> SentenceBERT 相似度取 top-k 节点      ≈ 精排（精细选最终的几个）
> ```
>
> 所以"粗排精排"这套成熟的工业方法论，很多思路可以直接搬来改进 G-Refer 的检索器。下面 5 篇，就是沿这条线挑的。

## 1. COLD：把粗排从"廉价替身"升级成"轻量版精排"

- **链接**：https://arxiv.org/abs/2007.16122 ｜ [PDF](https://arxiv.org/pdf/2007.16122)
- **出处**：阿里巴巴，DLP-KDD 2020

**这是哪个领域、在做什么？**
这是"粗排"这个子方向的奠基论文，几乎后面所有讲粗排的论文都会引它。它研究的问题是：**粗排到底能做多复杂？**

**在它之前是怎么做的（以及局限）：**
过去粗排为了快，被迫用最简单的**双塔模型**——用户一座塔、商品一座塔，各自算出一个向量，最后只做一次**向量内积**得分。这么做的好处是商品向量能提前算好存起来，线上超快；坏处是**用户特征和商品特征从头到尾没机会"交叉"**（比如"这个人 + 这件商品"这种组合信号完全用不上），所以粗排又糙又不准，只能当精排的"廉价替身"。

**这篇的创新点：**
1. **打破"粗排必须是简单模型"的教条**：作者说粗排也可以用带交叉特征的复杂深度模型，关键是把它"控制在算力预算内"。
2. **算力感知的联合优化**：它不是单纯追求模型更准，而是**同时优化"模型效果"和"算力开销"**这两个目标——把算力当成一个可以显式权衡的变量，在预算内尽量做复杂。
3. **软硬件协同的工程加速**：配合一堆推理加速技巧（低精度计算、并行等），把复杂模型压到线上能扛得住。

**大白话：**
以前大家觉得"海选嘛，随便看看就行，用最便宜的模型"。COLD 说：不对，海选也可以请个像样的裁判，只要你规定好"每人最多花 3 秒"这个预算，在预算内把裁判做到尽可能强。核心心态转变：**粗排不是"精排的缩水版"，而是"戴着算力镣铐跳舞的精排"。**

**跟 G-Refer 的关系：**
G-Refer 里"检索多少条路径、多少个节点"（论文里 k=2）本质就是一次算力-效果权衡——查得多解释可能更全，但更慢更贵。COLD 给了你一个正式的视角去讨论这个取舍，也支持你论证"我的剪枝+检索该做多重"。

## 2. Ranking Consistency：粗排真正该优化的，是"和精排保持一致"

- **链接**：https://arxiv.org/abs/2205.01289 ｜ [PDF](https://arxiv.org/pdf/2205.01289)
- **出处**：某大型中文电商平台（论文里没点名，实验在其上做）

**这是哪个领域、在做什么？**
还是粗排，但换了个更根本的问题：**粗排到底该拿什么当训练目标？** 大家习惯让粗排自己的 AUC（准确率）越高越好，这篇说：**方向错了。**

**问题背景：**
多阶段漏斗有个致命隐患——**上一阶段一旦把某个候选淘汰，下一阶段再也没机会捞回来**。所以粗排哪怕自己排得"看起来很准"，只要它把精排本来会选的好货提前砍掉了，整条链路就废了。也就是说：粗排的好坏，不该看它自己，而该看它**跟精排的排序合不合得上**。

**这篇的创新点：**
1. **形式化定义了"排序一致性"**：粗排的候选顺序，和精排最终的顺序，到底对不对得上。
2. **提出 RCS（Ranking Consistency Score）指标**：专门量化这种一致性，作者论证它比传统的"看单阶段准不准"更能反映整条链路的真实效果。
3. **给出提升一致性的方法**：从"样本怎么选"和"损失怎么设计"两方面入手（思路上类似知识蒸馏——让精排当老师、粗排当学生，粗排去模仿精排的打分）。

**大白话：**
海选评委的任务，不是自己审美有多高级，而是**别把决赛评委想要的选手在海选就刷掉**。衡量海选好坏的正确尺子，是"海选的排名和决赛的排名像不像"，而不是"海选评委自己打分准不准"。

**跟 G-Refer 的关系（很贴）：**
G-Refer 的路径检索器，做的正是"用一个**可解释的简单模型**（路径打分）去逼近一个**强的黑盒模型**（已训练好的 GNN）"——这跟"学生粗排模仿老师精排"是同一个套路。这篇的**一致性框架和 RCS 指标**，能给你一个现成的说法和度量：路径检索器选出的 top-k 边，到底和 GNN 的真实决策一不一致（你的 PaGE-Link 掩码扰动其实就在测这个"重要性一致性"）。

## 3. GNRR：在图上让候选"互相商量"再排序

- **链接**：https://arxiv.org/abs/2406.11720 ｜ [网页版](https://arxiv.org/html/2406.11720v1)
- **全称**：Graph Neural Re-Ranking via Corpus Graph

**这是哪个领域、在做什么？**
这是**信息检索里的"重排(re-ranking)"**方向（技术上和推荐精排相通）。它问的问题是：给候选排序时，能不能**考虑候选之间的关系**，而不是每个候选单独打分？

**在它之前是怎么做的（以及局限）：**
传统重排是"逐个打分"——每个文档单独和 query 算一个相关性分，然后按分排序。问题是：**候选之间是有关系的**（比如两篇文档讲的是同一件事），单独打分完全看不到这层关系，容易漏判。

**这篇的创新点（三步）：**
1. **初检**：先用便宜的 BM25 捞出 top-1000 候选。
2. **建"语料图"**：离线把文档做成一张图——节点是文档，边表示"两篇文档语义相似"。
3. **图上重排**：对每个 query，从大图里抽出"这次候选组成的小子图"，用 **GNN 让候选之间传递信息、互相影响**，再结合每个候选自己的相关性分，给出最终排序。

核心洞见：**候选彼此的分布关系里藏着信息，单点打分器抓不到**，用图能抓到，在 TREC-DL19 上相对提升约 5.8%。

**大白话：**
以前给一堆候选打分，像老师分别面试每个学生、互不干扰。GNRR 说：让学生们先在一个小组里讨论一下、互相参照，老师再打分——因为"这个候选相对于其他候选好不好"往往比"它自己绝对好不好"更有用。

**跟 G-Refer 的关系（最直接的缝合点）：**
G-Refer 现在选 top-k 节点，是**每个候选各自和目标算相似度**（SentenceBERT 内积），选出来的几个节点**彼此没有任何交互**。可以借 GNRR 的思路：**把选出的 top-k 用户/商品节点再组成一张小图，跑一轮 GNN 让它们互相校正**，去掉冗余、补齐互补信息，再喂给 LLM。这可能比现在的"独立 top-k"更干净、更少重复。

## 4. CHEST：为每次推荐抽一张"专属子图"

- **链接**：https://arxiv.org/abs/2106.06722 ｜ [PDF](https://arxiv.org/pdf/2106.06722)
- **全称**：Curriculum Pre-Training Heterogeneous Subgraph Transformer for Top-N Recommendation

**这是哪个领域、在做什么？**
这是**基于知识图谱/异构图的推荐**方向。"异构图"= 图里有多种类型的节点和边（用户、商品、品牌、导演……多种关系交织）。它要解决的问题是：这种大图里**信息太杂**，很多跟当前这次推荐根本没关系，怎么只挑出有用的那部分？

**在它之前是怎么做的（以及局限）：**
以前要么把整张异构图一股脑喂进模型（噪声大、算不动），要么只用简单的路径（信息不够）。缺一个"精准圈出相关范围"的办法。

**这篇的创新点：**
1. **交互相关的异构子图**：不为整张图建模，而是**为每一个"用户-商品"交互，单独抽出一张相关的小子图**（包含连接它俩的关键路径和实体），把范围收窄到真正相关的部分。
2. **异构子图 Transformer**：用 Transformer 编码这张子图的结构 + 路径语义（多 slot 序列表示）。
3. **课程预训练（curriculum）**：像上学一样**由易到难**分阶段训练——先学简单的通用图语义，再学复杂的推荐任务。数据稀疏时优势特别明显。

**大白话：**
知识图谱像一张巨大的关系网，但推荐"张三会不会喜欢这部电影"时，网里 99% 的信息（比如别的无关明星、无关商品）都是噪声。CHEST 的做法是：**每次只把和"张三↔这部电影"直接相关的那一小片关系网剪下来单独看**，而且训模型时先教简单的、再教难的，循序渐进。

**跟 G-Refer 的关系（对应你的创新点）：**
你在 G-Refer 笔记里写过——原论文**故意不选子图粒度**（怕搜索空间大、信息复杂），而你想"套 K-ragrec 的方法尝试做**子图级检索**"。CHEST 正好给了一个成熟范式：**怎么把"交互相关的子图"抽出来、怎么编码里面的路径语义**。你要做子图版 G-Refer，这篇是最直接的对标对象。

## 5. RankFlow：粗排精排别各练各的，串成一条"流"一起训

- **链接**：https://dl.acm.org/doi/10.1145/3477495.3532050 ｜ [PDF](https://www.ruizhang.info/publications/SIGIR%202022%20RankFlow%20-%20Joint%20Optimization%20of%20Multi-Stage%20Cascade%20Ranking.pdf)
- **出处**：SIGIR 2022

**这是哪个领域、在做什么？**
研究**整条多阶段漏斗的联合训练**。它问：召回、粗排、精排这几个模型，一直都是**各训各的**，这样真的好吗？

**问题背景（阶段偏差 Stage-Specific Bias）：**
每个阶段看到的数据分布都不一样——粗排看到的是召回筛剩下的，精排看到的是粗排筛剩下的。各自单独训练，就会**各自对着自己那批"筛过的数据"过拟合**，阶段之间目标不对齐，整条链路不是全局最优。这个毛病叫"阶段偏差"。

**这篇的创新点：**
1. **首个系统性的级联联合训练框架**：让各阶段的排序器不再孤立训练。
2. **信息像"流"一样双向传递**：每个阶段的训练数据由**前序阶段**生成（上游影响下游），而训练的指导信号不只来自日志，还来自**后序阶段**回传（下游反过来指导上游）。这样各阶段耦合成一条流，联合优化。

**大白话：**
一场三轮的选拔，以前海选、复赛、决赛三个评委组从不开会，各判各的，结果口味不统一、好苗子中途被误伤。RankFlow 让三组评委**坐下来对齐标准**：决赛评委把"我最终喜欢什么"的反馈传回给海选评委，海选评委据此调整自己该放谁进来——上下游互相通气，整体选出的人才最优。

**跟 G-Refer 的关系：**
G-Refer 是"路径检索器 + 节点检索器 → LLM"的级联，但三者**基本是分开训练**的，存在同样的错配风险：**检索器辛苦选出的 top-k，LLM 未必用得上**。RankFlow 的思路提示一个方向：**让下游 LLM 的生成质量信号回传，去指导上游检索器该选哪些路径/节点**，做端到端/联合优化，让检索和生成对齐。

## 泛读建议（两轮筛选，对应你笔记里的读法）

**第一轮（看摘要，判断想不想深究）**
- 想先把"粗排/精排到底是什么、该优化什么"搞明白 → 先读 **① COLD + ② Ranking Consistency**。
- 想直接找"能动手改 G-Refer 结构"的点 → 直接看 **③ GNRR + ④ CHEST**（图 + 子图，最能缝）。

**第二轮（细看，锁定缝合方向）**
- 若走"**子图级检索**"这条创新路线 → 精读 **④ CHEST**（方法可对标）+ 参考 **③ GNRR**（候选互相交互）。
- 若走"**检索器与 LLM 联合训练/对齐**"路线 → 精读 **⑤ RankFlow** + **② Ranking Consistency**。
